**Попередня оцінка (MVP):** 7.0 / 10  
**Нова оцінка (Поточний стан):** **9.2 / 10** (Видатний прогрес від MVP до зрілого продукту)

## 1. Вступ та огляд виконаних робіт

З моменту первинного рев'ю (15 червня 2026 року) у проєкті **AutoEDA** було проведено масштабну та всебічну роботу над усуненням архітектурних і методологічних недоліків, розширенням функціональності та підвищенням безпеки. 

Проєкт успішно переріс рамки простого прототипу (MVP) і наразі є повноцінним desktop-застосунком для автоматизованого аналізу даних та машинного навчання на платформі macOS.

### Що було успішно впроваджено (на основі перевірки коду):
1. **Мульти-пайплайнова архітектура (Phase C & D):**
   - **Tabular Pipeline:** покращено завдяки 5-Fold крос-валідації, стратифікації розбиття для класифікації та детальному профайлінгу.
   - **Time Series Pipeline:** додано сезонну декомпозицію (тренд, сезонність, залишки), обчислення автокореляцій (ACF/PACF) та генерацію часових ознак (lags, rolling stats) з хронологічним валідаційним сплітом (останні 20%).
   - **NLP/Text Pipeline:** впроваджено побудову матриць TF-IDF, навчання Naive Bayes моделей, обчислення важливості слів, розрахунок довжини текстів та вивід статистичних гістограм.
   - **Image Pipeline:** реалізовано роботу з NPZ-архівами та папками зображень, автоматичний ресайз, класифікацію зображень через Flatten/ML, обчислення середнього зображення класу та виведення сітки прикладів (image grid) у SwiftUI через Base64.

2. **Підвищення якості ML-методології (Phase B):**
   - **Крос-валідація:** додано 5-Fold CV з розрахунком середнього результату та стандартного відхилення, що робить метрики надійними та стійкими до оверфіттингу.
   - **Baseline порівняння:** інтегровано навчання Dummy-моделей (`DummyClassifier` та `DummyRegressor`), що дозволяє об'єктивно оцінити приріст якості від використання ML (наприклад, порівняно зі стратегією найчастішого класу чи середнього значення).
   - **Матриця помилок (Confusion Matrix):** обчислюється для тестового та валідаційного наборів і візуалізується у вигляді інтерактивної теплової карти (Heatmap) в UI.
   - **Аналітика кривих:** додано генерацію точок для ROC-кривих, Precision-Recall кривих, а також калібрувальних кривих (Calibration plots) та графіків залишків (Residuals).

3. **Data Profiling & Quality (Phase B/D):**
   - **Leakage Detection:** виявлення витоку даних (data leakage) на основі високої кореляції ознак із цільовою колонкою ($\ge 0.98$) або збігу назв колонок.
   - **Cleaning Recommendations:** автоматична видача рекомендацій щодо пропущених значень, константних колонок, викидів (outliers) та категорій високої кардинальності.
   - **Розширений огляд колонок:** інтегровано детальний профайлінг з квантилями (25%, 50%, 75%), мінімумами, максимумами, середнім відхиленням та розподілом категорій.

4. **Інтеграція з локальним штучним інтелектом (AI Grounding):**
   - **Передача контексту графіків:** при натисканні «Ask AI» на будь-якому графіку, його числові значення автоматично перетворюються на структуру даних і передаються в контекст Ollama, що унеможливлює галюцинації ШІ про дані.
   - **Покращений System Prompt:** AI тепер отримує структуровану інформацію про кореляції, якість даних, порівняння моделей та попередження про витік.

5. **Користувацький досвід (UX) та безпека:**
   - **Експорт звітів:** впроваджено генератор Markdown-звітів, який за допомогою локальної моделі Ollama створює якісний Executive Summary, Key Findings, а також додає таблиці метрик, описи графіків та семпли даних.
   - **Безпека облікових даних:** Kaggle API Keys та Hugging Face Tokens перенесено з небезпечного UserDefaults у системний **Keychain** через `KeychainService`.
   - **Керування процесами:** додано можливість безпечного скасування поточного аналізу (Cancel Analysis), що зупиняє Python-процес без зависання UI.
   - **Відтворюваність середовища:** додано скрипт `setup_env.sh` та файл `requirements.txt` для автоматичного створення `.venv` та інсталяції необхідних бібліотек.

## 2. Наскільки добре проєкт виконує свою задачу

Застосунок **відмінно** виконує задачу експрес-аналізу та побудови первинних моделей (AutoML / EDA). 
Користувачеві більше не потрібно писати код для базової аналітики:
1. Він перетягує файл (CSV, TSV, Parquet, NPZ).
2. Бачить зручний табличний прев'ю з можливістю **вимкнути** зайві колонки (Exclude Columns).
3. Обирає тип датасету (Tabular, Time Series, NLP, Image) та конфігурує цільову колонку (Target) і тип задачі (Classification/Regression/Auto).
4. Отримує інтерактивний дашборд із розподілами, матрицею кореляцій, порівнянням моделей, кривими ROC/PR, матрицею помилок та порадами щодо очищення.
5. Може поспілкуватися з локальним AI про конкретні графіки або експортувати готовий Markdown-звіт.

Проєкт повністю вирішує проблему швидкої оцінки якості даних та побудови перших ML-моделей «в один клік».

## 4. Виявлені недоліки та їх обґрунтування

Попри значний стрибок у якості, проєкт все ще має архітектурні та технічні компроміси, які варто враховувати:

### 1. Монолітний та перевантажений файл `analyze.py` (Об'єм ~160 KB, >3500 рядків)
* **В чому проблема:** Весь Python-бекграунд (завантаження даних, очищення, профайлінг, NLP, картинки, часові ряди, моделювання, розрахунок кривих, автокореляції та формування JSON-відповіді) знаходиться в одному гігантському файлі.
* **Чому це недолік:** Файл надзвичайно важко підтримувати, тестувати та розширювати. Будь-яка зміна в одній частині коду (наприклад, у логіці обробки картинок) може випадково зламати логіку аналізу часових рядів. 
* **Архітектурне обґрунтування:** Розробники залишили це в одному файлі для зручності пакування в Xcode-бандл (Swift легше знаходити один ресурс `analyze.py` через `Bundle.main.url` без потреби налаштовувати відносні шляхи до python-модулів). Проте для продукту промислового рівня це вважається "code smell".

### 2. Завантаження всього датасету в оперативну пам'ять (In-Memory з використанням Pandas)
* **В чому проблема:** Код використовує `pd.read_csv`, `pd.read_parquet` тощо, завантажуючи весь файл у пам'ять цілком.
* **Чому це недолік:** При роботі з великими датасетами (наприклад, CSV файли від 1 ГБ і більше) застосунок може викликати критичне споживання пам'яті (Out-Of-Memory) та аварійно завершити роботу Python-процесу, що призведе до падіння аналізу в SwiftUI.
* **Обґрунтування:** Немає механізму розумного семплювання (Smart Sampling) або використання більш сучасних бібліотек на кшталт `Polars` або `DuckDB` для великих даних.

### 3. Відсутність експорту коду пайплайна або навченої моделі
* **В чому проблема:** Застосунок навчає моделі, знаходить найкращу та показує її метрики. Проте користувач не може експортувати саму навчену модель (наприклад, у форматі `.joblib` або `.pkl`) або отримати Python-скрипт з кодом передопрацювання та навчання, щоб запустити його на іншому сервері.
* **Чому це недолік:** Для аналітика AutoEDA стає "чорною скринькою". Він знає, що модель хороша, але не може використати її в продакшні, що змушує його переписувати пайплайн вручну.

### 4. Обмеження складних моделей для Time Series та Feature Importance
* **В чому проблема:** Моделі часових рядів обмежені регресією на лагах (Linear/RandomForest). Немає класичних моделей прогнозування на кшталт `ARIMA` або `SARIMAX`. Також відсутній аналіз важливості ознак через `SHAP` (SHapley Additive exPlanations).
* **Чому це недолік:** Лагові фічі — це хороший бейзлайн, але для реального прогнозування трендів цього замало. Відсутність SHAP обмежує інтерпретованість складних моделей (користувач бачить лише загальну важливість ознак, але не напрямок їхнього впливу на прогноз).
* **Обґрунтування:** Бібліотеки на кшталт `prophet` або `shap` мають важкі бінарні залежності на C/C++ (на кшталт PyStan), які важко збираються на комп'ютерах користувачів (особливо на Apple Silicon M1/M2/M3) через `pip`. Усунення цих бібліотек — вимушений крок для стабільності середовища.

### 5. Відсутність інтерактивного застосування рекомендацій з очищення (Read-Only)
* **В чому проблема:** Панель "Auto-Cleaning Recommendations" показує чудові поради (наприклад, "Колонка `Age` має 15% пропусків -> заповніть медіаною"). Але користувач не може натиснути кнопку "Застосувати", щоб відкоригувати дані безпосередньо в застосунку перед моделюванням.
* **Чому це недолік:** Це змушує користувача йти в сторонній інструмент (Excel, Python), правити дані там, і знову завантажувати файл.

### 6. Відсутність локалізації інтерфейсу (Лише англійська мова)
* **В чому проблема:** Інтерфейс повністю англійськомовний.
* **Чому це недолік:** Для широкого кола локальних користувачів корисна була б українська локалізація, особливо враховуючи, що частина описів у попередніх аналізах була орієнтована на неї.

## 5. Пропозиції для нових фіч та покращень

Для виведення проєкту на рівень **9.8+/10** та створення комерційного продукту рекомендується впровадити такі покращення:

### 🚀 Фіча 1: Експорт коду пайплайну (Generate Code / Notebook)
Додати кнопку **"Export Pipeline Code"** після завершення аналізу.
* **Що вона робить:** Генерує автономний Python-скрипт або Jupyter Notebook (`pipeline.ipynb`), який містить:
  - Імпорт бібліотек.
  - Код завантаження файлу.
  - Точний етап передопрацювання (заповнення пропусків, One-Hot Encoding, скалювання).
  - Ініціалізацію та навчання найкращої моделі з тими ж гіперпараметрами.
* **Чому це круто:** Дає аналітику повну прозорість, дозволяє легко відтворити результат у хмарі та інтегрувати його у свій проєкт.

### 💾 Фіча 2: Збереження та експорт навченої моделі (Export Model)
Додати можливість зберегти серіалізований об'єкт моделі.
* **Що вона робить:** Зберігає найкращу модель разом із препроцесором у `.joblib` або `.zip` архів.
* **Чому це круто:** Дозволяє одразу використовувати модель для інференсу в реальних системах без перенавчання.

### ⚙️ Фіча 3: Інтерактивний Майстер Очищення (Interactive Cleaning Wizard)
Перетворити розділ рекомендацій на активний інструмент.
* Біля кожної рекомендації (наприклад: *Видалити колонку `ID`*) додати кнопку **"Apply"** (Застосувати).
* Користувач може обрати, які кроки виконати, після чого застосунок оновить внутрішній датасет та проведе повторний аналіз на очищених даних.

### 📊 Фіча 4: Smart Sampling для великих датасетів (Large Data Mode)
Захистити застосунок від падінь через нестачу пам'яті.
* Якщо розмір файлу перевищує 150 МБ, пропонувати користувачеві режим **"Smart Sampling"** (наприклад, читати випадкові 100,000 рядків або перші/останні N рядків із підтриманням пропорцій класів).
* Це збереже швидкість аналізу та убезпечить систему від OOM-помилок.

### 💬 Фіча 5: Інтеграція легких статистичних моделей для Time Series
Замість важкого Prophet додати класичний **Holt-Winters** або **Exponential Smoothing** із бібліотеки `statsmodels` (вона вже є в `requirements.txt`).
* Це дозволить будувати реальні прогнози (forecast) на майбутнє з довірчими інтервалами, не перевантажуючи систему важкими залежностями.

### 🌐 Фіча 6: Локалізація застосунку
Додати повну підтримку української мови для UI елементів та системних підказок (Localization String Tables).

## 6. Висновок

Проєкт **AutoEDA** продемонстрував колосальний прогрес. Розробники успішно усунули критичні зауваження щодо методології навчання моделей та безпеки (перенесення паролів у Keychain, впровадження крос-валідації, Dummy baseline, Confusion Matrix). Додавання трьох нових типів даних (картинки, тексти та часові ряди) виводить інструмент у категорію універсальних EDA-клієнтів.

**Поточна оцінка 9.2/10** відображає високий ступінь готовності продукту як локального інструменту аналітика. Подальший розвиток у напрямку генерації коду, інтерактивного очищення даних та оптимізації роботи з великими файлами дозволить AutoEDA стати лідером серед легких AutoML рішень для macOS.
